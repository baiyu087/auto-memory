# Promote Lessons

将项目 `lessons.md` 中的教训条目复审，选择重要的规则固化到 CLAUDE.md。

## 执行步骤

### 第一步：读取教训列表

读取当前项目的 lessons.md 文件：
- 路径：`~/.claude/projects/<project-id>/memory/lessons.md`
- project-id 由当前工作目录路径转换（`/` 替换为 `-`，Windows 的 `:` 删除）

### 第二步：分析并分类展示

对每条教训，判断其适合放在哪一级：

**用户级 (`~/.claude/CLAUDE.md`)** — 跨所有项目通用的规则：
- 沟通风格、称呼偏好
- 通用工作流约束（如"修改前先确认"）
- 跨项目一致的工具使用习惯

**项目级（项目根目录 `CLAUDE.md`）** — 仅适用于当前项目的规则：
- 项目特有的文件结构约定
- 特定技术栈的使用规范
- 项目专属的业务逻辑约束

以紧凑列表形式一次展示所有教训，每条一行，标注建议级别：

```
📋 教训复审 · 共 N 条
══════════════════════════════════════

#1 · 纠正(C) · 建议：用户级
内容摘要（超长时截断）...

#2 · 工具(T) · 建议：删除
内容摘要...

...

══════════════════════════════════════
请回复处理方式，例如：`1=U, 2=D, 3=P`
- U 用户级　P 项目级　S 跳过　D 删除

或直接说 "按建议执行"
```

**建议级别判断依据：**
- 用户级：沟通风格、通用工作流约束、跨项目工具习惯
- 项目级：项目特有文件结构、特定技术栈规范、项目专属业务约束
- 删除：一次性技术调试细节、已内化到代码中的实现知识、验证记录

### 第三步：用户选择

等待用户回复，支持以下格式：
- 逐条指定：`1=U, 2=U, 3=D, 4=D`
- 快捷指令：`按建议执行`

### 第四步：执行写入

根据用户选择：

**写入前去重检查（必须执行）：**
1. 读取目标 CLAUDE.md 的完整内容
2. 对每条待写入的教训，判断是否已存在语义相近的规则（不要求完全匹配，含义相同即视为重复）
3. 若已存在：跳过写入，在汇报中标注"已存在，跳过"
4. 若不存在：继续写入

**写入用户级 CLAUDE.md（`~/.claude/CLAUDE.md`）：**
- 找到语义最相关的章节追加
- 若无合适章节，在文末新增章节

**写入项目级 CLAUDE.md（项目根目录）：**
- 同上，找合适章节追加
- 若文件不存在，创建并写入

**跳过：** 保留在 lessons.md，下次复审时仍会出现

**删除：** 从 lessons.md 中移除该条目（技术细节类教训、已内化到代码中的实现知识、验证记录）

### 第五步：更新复审时间

写入完成后，更新 lessons.md 中的元数据：
```
<!-- meta: last_reviewed: YYYY-MM-DD -->
```

### 第六步：确认汇报

汇报执行结果：
- 写入用户级 N 条
- 写入项目级 N 条
- 已存在跳过 N 条（注明是哪条）
- 跳过 N 条
- 删除 N 条
